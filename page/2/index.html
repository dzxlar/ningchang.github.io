【Java源码】String字符串

发表于 2017-07-28 | 分类于 String

String特性

  • 通过final修饰(Java7以后说明：final类方法永远不会被重写)

    public final class String{...}

  • String为不可变性（不可通过正常手段修改值）

    - 通过substring方法作为代表
        public String substring(int beginIndex) {
            if (beginIndex < 0) {
                throw new StringIndexOutOfBoundsException(beginIndex);
            }
            int subLen = value.length - beginIndex;
            if (subLen < 0) {
                throw new StringIndexOutOfBoundsException(subLen);
            }
            //通过重新创建String对象修改值
            return (beginIndex == 0) ? this : new String(value, beginIndex, subLen);
        }

    不可变的好处

  1. 可以实现多个变量引用JVM内存中的同一个字符串实例。
  2. 安全性
  3. 性能，String大量运用在哈希的处理中，由于String的不可变性，只计算一次哈希值，然后缓存在内部，后续直接取值

设计成不可变后，可以多个变量引用JVM上同一块地址,可以节省内存空间，相同的字符串不用重复占用Heap区域空间

拼接字符串

  1. 不推荐”+”号拼接

    每次循环都会创建一个StringBuilder，在末尾再调用toString返还回去

  2. 推荐append方法

  3. StringBuffer线程安全、StringBuilder非线程安全

Hive学习笔记（一）

概念

Hive是一个基于Apache Hadoop的数据仓库。对于数据存储与处理，Hadoop提供了主要的扩展和容错能力。

对于大量的数据，使得数据汇总，查询和分析更加简单。它提供了SQL，允许用户更加简单地进行查询，汇总和数据分析。

同时，Hive的SQL给予了用户多种方式来集成自己的功能，然后做定制化的查询，例如用户自定义函数

结构

  • 数据库
    • 相当于命名空间，避免表，视图，分区，列等等的命名冲突
  • 表
    • 相同数据库的同类数据单元
  • 分区
    • 每个表可以有一个或多个用于决定数据如何存储的分区键
  • 桶
    • 每个分区的数据，基于表的一些列的哈希函数值，又被分割成桶

数据类型

  • Integer
    • tinyint、smallint、int、bigint
  • 布尔
    • boolean
  • 浮点数
    • float
    • double
  • 定点数
    • decimal
  • 字符串
    • string
    • varchar
    • char
  • 日期和时间
    • timestemp
    • date
  • 二进制
    • binary
  • 复杂类型
    • 结构体类型(Stuct)
      • 列a{b int,c,int},通过列.属性访问(例如：a.b,a.c）
    • Map键值对类型
      • 通过[‘colName’]访问
    • 数组
      • 通过[‘n’]访问数组

内置运算和函数

参考文档

  • 关系运算
  • 数学运算
  • 逻辑运算
  • 复杂类型运算

提供基本SQL

  • where筛选条件
  • select查询指定列
  • 2表join查询
  • 多个group by列上使用聚合
  • 可以存储擦讯结构到另一个表
  • 可以下载表内容到本地
  • 可以存储查询结果到hadoop分布式文件系统目录
  • 可以管理表和分区
  • 可以使用自定义脚本

加载数据

用户可以创建一个“外部表”来指向一个特定的HDFS路径。用这种方法，用户可以使用HDFSput或copy命令，复制一个文件到指定的位置，并且附上相应的行格式信息创建一个表指定这个位置

例如，如果文件/tmp/pv_2016-06-08.txt包含逗号分隔的页面访问记录。这需要以合适的分区加载到表page_view

CREATE EXTERNAL TABLE page_view_stg(viewTime INT, userid BIGINT,
        page_url STRING, referrer_url STRING,
        ip STRING COMMENT 'IP Address of the User',
        country STRING COMMENT 'country of origination')
COMMENT 'This is the staging page view table'
ROW FORMAT DELIMITED FIELDS TERMINATED BY '44' LINES TERMINATED BY '12'
STORED AS TEXTFILE
LOCATION '/user/data/staging/page_view';

hadoop dfs -put /tmp/pv_2016-06-08.txt /user/data/staging/page_view

联结查询

LEFT OUTER,RIGHT OUTER或FULL OUTER
INSERT OVERWRITE TABLE pv_users
SELECT pv.*, u.gender, u.age
FROM user u JOIN page_view pv ON (pv.userid = u.id)
WHERE pv.date = ‘2008-03-03’;

为了检查key在另外一个表中是否存在，用户可以使用LEFT SEMI JOIN

INSERT OVERWRITE TABLE pv_users
SELECT u.*
FROM user u LEFT SEMI JOIN page_view pv ON (pv.userid = u.id)
WHERE pv.date = '2008-03-03';

多表联结

INSERT OVERWRITE TABLE pv_friends
SELECT pv.*, u.gender, u.age, f.friends
FROM page_view pv JOIN user u ON (pv.userid = u.id) JOIN friend_list f ON (u.id = f.uid)
WHERE pv.date = '2008-03-03';

聚合

INSERT OVERWRITE TABLE pv_gender_sum
SELECT pv_users.gender, count (DISTINCT pv_users.userid)
FROM pv_users
GROUP BY pv_users.gender;

两个聚合函数不能同时用DISTINCT作用于不同的列，以下情况是可以的（DISTINCT作用于相同列）

INSERT OVERWRITE TABLE pv_gender_agg
SELECT pv_users.gender, count(DISTINCT pv_users.userid), count(*), sum(DISTINCT pv_users.userid)
FROM pv_users
GROUP BY pv_users.gender;

  1. 降低资源消耗
  2. 提高响应速度
  3. 提高线程的可管理性

线程池步骤

  1. 提交任务
  2. 核心线程池未满、创建线程执行任务
  3. 核心线程池已满、队列未满，存储在队列中
  4. 队列已满、创建线程执行任务

CountDownLatch

允许一个或多个线程等待其他线程完成操作

具体实例

解析Excel中的每个Sheet数据。考虑通过多线程解析每个Sheet数据。所有线程解析完毕后，才算解析完成

方法一: join()

Thread parse1 = new Thread(new Runnable() {

            @Override
            public void run() {
                System.out.println(1);
            }
        });
        Thread parse2 = new Thread(new Runnable() {

            @Override
            public void run() {
                System.out.println(2);
            }
        });
        parse1.start();
        parse2.start();
        parse1.join();
        parse2.join();
        System.out.println("all");

join用于当前执行线程等待join线程执行结束。原理：不停检查join线程是否存活。如果join线程存活则让当前线程永远等待.wait(0)会让线程永远等待。直到join线程中止后，线程的notifyall()方法被调用

方法二: CountDownLatch

final int N = 2;
CountDownLatch latch = new CountDownLatch(N);
new Thread(new Runnable() {

     @Override
     public void run() {
         System.out.println(1);
         latch.countDown();
         System.out.println(2);
         latch.countDown();
     }
 }).start();
 latch.await();
 System.out.println(3);

CountDownLatch传入int类型的作为计数器。等待N个点完成

每次调用countDown()方法时N就会减少1.await()会阻塞当前线程，直到N变成0

countDown可以用在任何地方。所以可以代表N个线程，也可以代表1个线程内执行的N个步骤

如果其中某个解析sheet线程处理较慢。不让主线程一直等待，可以使用带时间的await()的方法。等待特定时间后就会不在阻塞当前线程

注意：1. 计数器必须大于等于0。 2. 当N=0时，await()不会阻塞当前线程。3.CountDownLatch不能重新初始化或者修改内部计数器的值。 3.一个线程调用countDown方法happens-before，另外一个线程调用await

Cyclicbar

让一组线程到达一个屏障（同步点）被阻塞。直到最后一个线程达到屏障时，线程放行，所有被屏障拦截的线程才会继续执行

final int N = 2;
CyclicBarrier barrier = new CyclicBarrier(N);
new Thread(new Runnable() {
     @Override
     public void run() {
         try {
             barrier.await();
         } catch (InterruptedException e) {
             e.printStackTrace();
         } catch (BrokenBarrierException e) {
             e.printStackTrace();
         }
         System.out.println(1);
     }
 }).start();
 try {
     barrier.await();
 } catch (BrokenBarrierException e) {
     e.printStackTrace();
 }
 System.out.println(2);

由于主线程和子线程的调度通过cpu决定执行的。所以先后顺序可能是1,2或者是2,1

如果new CyclicBarrier(3)则线程会永远等待。因为没有第三个线程执行await()方法。所以会一直等待。

CyclicBarrier(int parties,Runnable barrierAction),用于在线程达到屏障时候，优先执行barrierAction。处理更复杂的业务场景

    final int N = 2;
       CyclicBarrier barrier = new CyclicBarrier(N,new ThreadA());
       new Thread(new Runnable() {
            @Override
            public void run() {
                try {
                    barrier.await();
                } catch (InterruptedException e) {
                    e.printStackTrace();
                } catch (BrokenBarrierException e) {
                    e.printStackTrace();
                }
                System.out.println(1);
            }
        }).start();
        try {
            barrier.await();
        } catch (BrokenBarrierException e) {
            e.printStackTrace();
        }
        System.out.println(2);;                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      
}

static class ThreadA implements Runnable{

    @Override
    public void run() {
        System.out.println(3);
    }

}

CountDownLatch和CyclicBarrier的区别

1、 CountDownLatch计数器只使用1次，而CyclicBarrier的计数器可以通过reset()重置

  1. CyclicBarrier可以获得阻塞线程数量。isBroken()判断是否中断线程

Semaphore

控制同时访问资源的线程流量（限流）。相当于每次只允许特定数量的人进入，再根据里面出来多少的数量决定再次进入多少，以此类推。

final int N = 10;
Semaphore semaphore = new Semaphore(N);
ExecutorService threadPool = Executors.newFixedThreadPool(30); 
for (int i = 0; i < 30; i++) {
    threadPool.execute(new Runnable() {
     @Override
     public void run() {
         try {
             semaphore.acquire();
             System.out.println("save data");
         } catch (InterruptedException e) {
             e.printStackTrace();
         }
         semaphore.release();
     }
 });
}
threadPool.shutdown();

【多线程】线程中的原子操作基本概念

发表于 2017-06-23 | 分类于 多线程 ， Atomic

基本概念

当一个请求在多线程环境下涉及到更新变量并且是原子性增长的，在线程非安全情况下可能会获得不同的值（例如：用户的点赞 A用户（A线程）点赞+1，B用户（B线程）点赞+1，最终得到的结果可能是2，不是3)

为了保证程序在多线程环境下计数正确。Java引入atomic包（包内的原子操作类为更新一个变量提供一种高效、简单、线程安全的方式）

因为变量类型有很多种所以在Atomic包内提供了13种原子操作类;属于4种类型的原子更新方式。分别是原子更新基本类型、原子更新数组、原子更新引用、原子更新属性(字段)

Atomic包里的类基本都是使用Unsafe实现的包装类

原子更新基本类

  • AtomicInteger:原子更新整型
  • AtomicBoolean:原子更新布尔类型
  • AtomicLong:原子更新长整型

原子更新数组

  • AtomicIntegerArray 原子更新整型数组的元素
  • AtomicLongArray 原子更新长整型数组的元素
  • AtomicReferenceArray 原子更新引用类型数组的元素

原子更新引用类型

  • AtomicReference 更新引用类型
  • AtomicReferenceFieldUpdater 更新引用类型的字段
  • AtomicMarkableReference 更新带有标记位的引用类型

【源代码解析】JDK1.8——ConcurrentHashMap

发表于 2017-06-22 | 分类于 Java

JDK1.8的改进

  1. 采用transient volatile HashEntry< K,V > table保存数据。table数组元素作为锁，从而实现了对每一行数据进行加锁，进一步减少并发冲突的概率

  2. 将原先table数组＋单向链表的数据结构，变更为table数组＋单向链表＋红黑树的结构。采用单向链表，查询某个节点的时间复杂度为O(n)；对于个数超过8(默认值)的列表，jdk1.8中采用了红黑树的结构，那么查询的时间复杂度可以降低到O(logN)，以提高性能

   public ConcurrentHashMap(int initialCapacity,
                         float loadFactor, int concurrencyLevel) {
    if (!(loadFactor > 0.0f) || initialCapacity < 0 || concurrencyLevel <= 0)
        throw new IllegalArgumentException();
    if (initialCapacity < concurrencyLevel)   // Use at least as many bins
        initialCapacity = concurrencyLevel;   // as estimated threads
    long size = (long)(1.0 + (long)initialCapacity / loadFactor);
    int cap = (size >= (long)MAXIMUM_CAPACITY) ?
        MAXIMUM_CAPACITY : tableSizeFor((int)size);
    this.sizeCtl = cap;
 }
